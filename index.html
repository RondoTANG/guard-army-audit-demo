视频号/抖音直播 AI 审核开发指南说明页

AI 审核优化 (Super Agent) 多模态接口输出文档规范 · 最终输出 JSON 结构定义
v2.5 全栈对齐规范

后端/前端开发同学你好，此指南详细阐述了 AI 审核工作流的 Start 入参矩阵以及最终输出结果变量 finalresult 的内部 JSON 结构与多维判定指标。请以此规范进行业务对账与前端状态直透。

一、 工作流全局输入参数说明 (Start 节点入参)

当业务系统发起一单新的互动审核调度请求时，需向 HiAgent 工作流 Start 节点传入以下全局入参矩阵：

输入变量名 数据类型 必填与默认值 业务含义与配置说明
images Array<String> 必填 用户提交上传的待审图片二进制流或绝对网络 URL 数组，将送入多模态大模型循环进行分类提取。
checktype String 必填 指定审核的作业平台模型。枚举值：fwcm (微信视频号，强制检查朋友圈转发与评论) 或 cm (抖音直播，仅查验评论数)。
cmnum Number 选填 (默认 1) 作业配置设定的最低评论合规张数门槛（如：设定为 2，则合格评论截图数达 2 张才算完全达标）。
user_nickname String 选填 用户在平台绑定的发帖人微信/抖音昵称，用于验证朋友圈头部发帖账号是否属于本人。
live_room_name_input String 选填 指定查验的目标直播间或品牌车系名称（如“东风汽车”、“猛士科技”），支持大模型模糊容错核验。
time_interval Number 选填 (默认 2) 评论间隔时间间隔设定值（单位：分钟）。若前后两张图相似且时间间隔少于此数值，则触发作弊雷同拦截。

二、 总体输出结构 (根节点 Key)

finalresult 解析后的根级对象包含以下核心字段，主要用于整单任务的定性和提示：

字段名 类型 业务说明与判定准则
task_passed Boolean 核心业务依据。判定整单工单是否总体通过。只要 comments_passed_count > 0 且视频号转发图有效，即为 true；若有效评论数为 0 或视频号转发图校验失败，则直接强制记为 false。
forwarding_passed Boolean / Null 转发图是否通过。仅在视频号任务 (fwcm) 中有效，若为 true 表示转发图合格；抖音任务 (cm) 为 null。
comments_passed_count Number 有效评论图通过张数。经过防伪去重计算后，最终判定合格录入核账池的直播间截图数。
required_comments Number 要求的评论张数门槛。对应作业配置设定的目标要求值 cmnum。
final_valid_count Number 兼容历史结算系统的合规数计件字段，数值等同于 comments_passed_count。
message String 给审核人员及 C 端用户展示的精准结算提示语。如：“转发已通过；评论去重通过 1 张 (需 2 张，按张计分部分达标)” 或 “有效评论 0 张 (需 2 张，审核不通过)”。
detailed_results Array<Object> 包含用户上传的每一张图片的详尽解析、特征提取、判罚依据与去重指纹数组（见下文）。
conversation_id String 智能体调用记录ID：该工单AI审核调用在智能体(Agent)底层的唯一轨迹跟踪ID，方便追溯和排障追踪。

三、 图片明细解析对象 (detailed_results 数组元素)

该数组内的每个 Object 代表单张图片的处理结果。为方便前端统一解析渲染，所有图片对象都会输出同一套全量 Key 矩阵（无关或未识别字段自动赋空值）。请严格根据 category 字段的值来分发对应业务逻辑模块。

🔹1. 通用基础字段 (所有图片对象有效)

字段名 类型 说明与枚举
index Number 图片传入接口时的原始数组下标（用于前端回显对齐与溯源）。
imagename String 图片的文件名或云存储 URL 标识。
category String 图片类型识别结论："moments"(朋友圈)、"live_stream"(直播间) 或 "invalid"(非业务相关图)。
status String 单图定性结论："passed" (合格通过) 或 "failed" (不合格/雷同作弊)。
reason String 判罚理由：解释为什么通过或被驳回（如“朋友圈验证通过”、“与图1内容极度相似，判为作弊”等）。
think String AI 视觉多模态大模型判定分类与提取特征时的内部链式推理过程日志。

🔹2. 朋友圈转发图独有业务 Key (category == "moments")

字段名 类型 说明与判定准则
match Boolean 兜底标识，朋友圈图片固定返回 true。
name_match_status String 发帖人昵称比对状态。枚举：MATCHED (完全一致), NOT_FOUND (未提取到昵称), MISMATCHED (不一致)。
user_nickname_extracted String 画面顶部提取到的发帖人实际微信昵称原文。
is_live_stream Boolean 是否带有“红色直播中/灰色直播结束”状态标识或底部视频号来源挂件条。
【拦截强校验】：后端验证代码已收紧，要求 match == true 且 is_live_stream == true 同时满足时，该截图的 status 才会判定为 passed。
live_status String 提取到的标签文字（如“直播中”或“直播已结束”）。
share_text String 用户发布朋友圈时的纯文本配文。
video_channel_name String 底部来源条的后缀名称（如“东风汽车”）。

🔹3. 直播间评论图独有去重与业务 Key (category == "live_stream")

字段名 类型 说明与判定准则
live_room_name_match Boolean 直播间主体是否达标：结合严格字符匹配与车企品牌（东风、猛士等）语义兜底比对得出的综合判定。
livename_match_status String 直播间账号名纯字面匹配结果：MATCHED, NOT_FOUND (空值提取), MISMATCHED。
live_room_name String 画面左上角实际识别到的账号名称原文。
user_comment_found Boolean 是否在公屏列表或底部第一人称输入框成功寻获对应绑定用户的有效评论。
name_match_status String 评论用户昵称匹配状态：MATCHED (实际评论昵称与输入 user_nickname 一致), NOT_FOUND (未找到评论), MISMATCHED (找到评论但昵称不一致)。
user_comment String 实际定位到的单条评论内容文本。
fingerprint_time String 防伪核心1：手机系统顶部通知栏的时间戳（提取纯分钟数用于间隔校验）。
fingerprint_comments String 防伪核心2：除本人发言外的公屏其余弹幕拼接字符串，用于算法判定内容雷同度。
fingerprint_visual String 防伪核心3：机位视角与悬浮飘窗挂件等视觉元素组合。
is_duplicate Boolean 防伪拦截结论：若为 true 表示该图被判定为作弊/重复上传截图，整单结算核收数量直接予以剔除。
debug_info String 防伪计算存证日志：输出具体比对分值（如“时差:0m, 评论相似度:1.0, 画面:0.85”），用作申诉举证。

🔹4. 无效截图输出 (category == "invalid")

说明与判定准则
当模型判定图片既非朋友圈又非直播间（如纯聊天对话页、纯系统桌面或第三方APP截图）时，将归类为 invalid。
此时 JSON 结构中仅保留全局基础字段：think, index, imagename, category。
系统接收到该类型后直接熔断判定为不合格（非目标任务图片），极大节省了无意义的业务字段解析与回显逻辑。

四、 评论图防伪去重算法与判定规则说明 (Deduplication Engine)

为了有效防范刷单人员利用连拍截图、盗图录屏或多机位位移等作弊手段骗取积分，工作流在多模态特征识别后，独立运行了一套严密的三维去重与防伪存证算法：

🛡️三维复合比对模型 (判定阈值)

系统会取当前待判定的直播间截图与前置已通过的合规图片池逐一进行交叉比对。若以下三项指标同时触发判定阈值，则强制标记该图为作弊雷同图 (is_duplicate = true)：

  • 1. 截屏时差过近 (time_diff < time_interval)：自动提取通知栏手机时间戳计算绝对差值。若差值小于作业设定的最小间隔要求（默认 2 分钟，如 10:00 与 10:01 仅差 1 分钟），判定为短时间连拍嫌疑。
  • 2. 公屏弹幕指纹雷同度高 (comments_sim > 0.8)：过滤掉用户本人的评论，将公屏其余路人弹幕拼接清洗后，利用序列哈希相似度计算文本差异。若重合度超过 80%，证明处于同场同一时段。
  • 3. 视觉空间构图高度相似 (visual_sim > 0.8)：对比主播出镜位置、座舱角度、悬浮飘窗挂件等结构化视觉标签。若视觉空间构图特征重合度超过 80%，证明未发生场景切换。
🚨 拦截结果与存证举证：一旦判定为作弊，单图结算打上 failed 并不计入总分。同时系统会在 debug_info 中完整记录证据链（例如：命中去重(与图1) - 时差:0m, 评论相似:1.00, 画面相似:0.85），为运营申诉排障提供无死角实锤数据。
